# 家庭买菜做饭助手 Wiki

## 1. 项目定位

### 1.1 产品一句话

**今天吃什么：输入家庭情况，自动告诉你买什么、做什么、怎么做。**

### 1.2 本质定义

这个项目不是普通菜谱工具，而是：

> **家庭买菜决策工具 + 一桌饭生成器 + 新手做饭执行助手。**

普通菜谱 App 的逻辑是：

```text
用户已经知道想做什么菜 → 搜索菜谱 → 看做法
```

本项目的逻辑是：

```text
用户不知道今天买什么、做什么、怎么做
↓
系统根据家庭情况、预算、深圳季节、常见食材、菜品库
↓
自动生成三套完整晚餐方案
↓
输出买菜清单、做饭顺序、详细做法
```

### 1.3 核心用户

主要面向：

```text
姐姐 / 宝妈 / 家庭做饭负责人
年轻家庭
情侣同居
新手做饭用户
下班后买菜做饭的人
有孩子/老人需要照顾饮食的人
深圳本地家庭
```

### 1.4 核心痛点

用户的三个主要难点：

```text
1. 不知道买什么菜
2. 不知道做什么菜
3. 不知道怎么做
```

扩展痛点：

```text
不知道怎么搭配一桌饭
不知道买多少
不知道几个人够不够吃
不知道菜市场/朴朴/沃尔玛该搜什么
不会安排几道菜的做饭顺序
怕漏买食材
怕菜价太贵
怕小孩/老人不适合吃
```

## 2. 产品核心功能

### 2.1 MVP 核心流程

第一版最小可行版本只做：

```text
用户输入家庭情况
↓
系统生成三套菜单
↓
自动生成买菜清单
↓
生成每道菜详细做法
↓
生成整桌菜做饭顺序
↓
提供替换建议
```

### 2.2 用户输入项

首页不要让用户填太多，第一版只保留关键问题：

```text
1. 几个人吃饭？
2. 有小孩/老人吗？
3. 预算多少？
4. 想吃清淡还是重口？
5. 今天想省事还是吃好点？
6. 有什么不吃？
7. 买菜方式：菜市场 / 楼下超市 / 朴朴 / 沃尔玛 / 盒马 / 美团买菜
8. 做饭时间：20分钟 / 30分钟 / 40分钟 / 1小时
```

高级选项后续再加：

```text
是否减脂
是否控糖
是否不吃辣
是否不吃海鲜
是否不吃牛羊肉
是否不吃香菜
是否只用电饭锅
是否只有电磁炉
是否有空气炸锅
是否要儿童友好
是否要老人清淡
```

### 2.3 三套菜单固定风格

每次推荐 3 套，不要给太多，否则用户更纠结。

推荐固定为：

```text
A. 省钱快手型
B. 营养均衡型
C. 孩子/老人友好型
```

如果没有小孩老人，第三套改成：

```text
C. 改善伙食型
```

也可以根据场景变化：

```text
A. 30分钟快手
B. 清淡汤水
C. 周末改善
```

### 2.4 每套菜单输出内容

每一桌菜必须包含：

```text
菜单名称
适合场景
菜品列表
预计花费
预计耗时
买菜清单
做饭顺序
每道菜详细做法
替换建议
为什么这么搭配
注意事项
```

## 3. 产品与普通菜谱 App 的区别

### 3.1 普通菜谱 App

```text
用户知道要做番茄炒蛋
↓
搜索番茄炒蛋
↓
看教程
```

### 3.2 本产品

```text
用户不知道今晚吃什么
↓
告诉系统几个人、预算、忌口、时间
↓
系统直接安排一桌饭
↓
告诉她买什么、买多少、怎么做、先做哪个
```

### 3.3 真正壁垒

不是"有很多菜谱"，而是：

```text
把菜品组合成适合一家人吃的一桌饭
根据人数、预算、口味、季节、做饭时间自动搭配
生成可执行买菜清单
生成多菜并行做饭顺序
根据深圳本地买菜场景推荐
```

## 4. 深圳场景设计

### 4.1 深圳用户特点

```text
1. 下班晚，做饭时间短
2. 年轻家庭多
3. 有孩子的小家庭多
4. 买菜渠道多：菜市场、朴朴、沃尔玛、盒马、美团买菜、楼下超市
5. 天气热，适合清淡、汤水、瓜类、快手菜
6. 租房厨房可能小，不适合复杂大菜
7. 广东口味偏清淡，但深圳人口混杂，也要支持川湘重口味
```

### 4.2 买菜渠道模式

- **菜市场**：新鲜便宜品类多。推荐鱼、青菜、瘦肉、排骨、鸡肉、豆腐、当季瓜果
- **朴朴**：日常配送快常见食材。推荐鸡腿、鸡蛋、番茄、土豆、菜心、生菜、豆腐、瘦肉
- **沃尔玛**：超市常见包装肉菜。推荐鸡蛋、番茄、土豆、胡萝卜、鸡腿、猪肉、青菜、豆腐
- **盒马**：品质高海鲜多半成品多。推荐清蒸鱼、虾、牛肉、半成品汤包、儿童餐
- **楼下超市**：临时买菜选择有限。推荐鸡蛋、番茄、青菜、豆腐、火腿、面条

输出时要给平台搜索词：

```text
菜心 → 菜心 / 广东菜心
鸡腿 → 琵琶腿 / 鸡腿肉 / 鸡全腿
瘦肉 → 前腿肉 / 里脊肉 / 瘦肉
```

## 5. 一桌饭生成规则

### 5.1 按人数搭配

- **2 人**：1 荤 + 1 素 + 1 汤/蛋 + 主食
- **3 人**：1 荤 + 1 半荤素 + 1 素 + 1 汤 + 主食
- **4 人**：2 荤 + 1 素 + 1 汤，或 1 硬菜 + 1 家常 + 1 蛋/豆腐 + 1 青菜 + 1 汤
- **5-6 人**：2-3 荤 + 2 素 + 1 汤 + 1 凉菜/蒸菜 + 主食

### 5.2 有小孩时

必须至少 1 道不辣软菜、1 道蛋类/豆腐/鱼/肉末；少骨少刺少油少辣。

推荐：番茄炒蛋、肉末蒸蛋、虾仁滑蛋、土豆焖鸡、清蒸鲈鱼、冬瓜肉丸汤、玉米胡萝卜排骨汤、豆腐肉末、番茄牛肉、南瓜蒸排骨。

### 5.3 有老人时

优先清淡、软烂、少油、汤水、鱼肉、豆腐、蒸菜、易咀嚼。

推荐：肉末蒸蛋、清蒸鱼、冬瓜肉片汤、豆腐肉末、瘦肉水、玉米胡萝卜排骨汤、清炒菜心、蒸南瓜。

### 5.4 减脂场景

优先鸡胸、虾、鱼、牛肉、豆腐、鸡蛋、大量蔬菜；避免油炸、重油、糖醋、肥肉、过多米饭。

## 9. 数据来源策略

### 9.1 数据分层

```text
第一层：菜品库      → 大模型批量生成 + 人工校验
第二层：食材季节库  → 大模型 + 人工校正 + 稳定时令规律
第三层：今日价格   → 公开菜价 + 去年同期 + 少量修正 + 用户反馈
```

### 9.2 第一版不接商超实时接口

沃尔玛/朴朴/盒马/美团买菜这类平台，通常没有适合个人开发者直接拿门店实时库存的公开接口。难拿、合规风险、反爬麻烦。

替代：去年同期公开数据 + 时令表 + 常见商超商品池 + 用户反馈 + 平台搜索词。

## 10. 去年数据复用方案

### 10.1 核心思路

```text
去年同期数据 = 基准菜池 / 基准价格 / 基准时令
今年数据 = 小幅修正 / 用户反馈 / 少量公开价格接口
```

### 10.2 历史基线表 DDL

```sql
create table historical_ingredient_baseline (
  id uuid primary key default gen_random_uuid(),
  ingredient_name text not null,
  city text default '深圳',
  month int not null,
  source text,
  avg_price numeric,
  price_min numeric,
  price_max numeric,
  unit text,
  season_score int,
  availability_score int,
  recommend_score int,
  updated_at timestamp default now()
);
```

### 10.3 当前修正表 DDL

```sql
create table current_adjustment (
  id uuid primary key default gen_random_uuid(),
  ingredient_name text,
  city text default '深圳',
  date date,
  price_adjustment numeric,
  availability_adjustment int,
  reason text,
  source text
);
```

### 10.4 今年估算价公式

```text
今年估算价 = 去年同期价格 × 当前修正系数
```

产品里只显示：便宜 / 正常 / 偏贵 / 很贵。

## 11. 今日推荐食材池

### 11.1 作用

```text
决定今天优先推荐哪些食材
避免推荐太贵、不当季、不好买的菜
让菜单更贴近深圳本地生活
```

### 11.2 每日生成逻辑

```text
1. 读取当前月份
2. 找去年同月深圳高推荐食材
3. 结合华南/深圳时令表
4. 结合当前价格修正
5. 去掉太贵/不好买/不适合家庭的食材
6. 生成今日推荐食材池
7. 菜品推荐时优先使用这些食材
```

### 11.3 推荐分公式

```text
推荐分 = 时令分 × 0.35
       + 去年同期常见度 × 0.25
       + 今年价格修正 × 0.20
       + 家庭适配度 × 0.20
```

## 12. 大模型能做什么

### 12.1 适合做的

1. 理解用户自然语言需求 → 结构化 JSON
2. 批量生成菜品库
3. 从候选菜中组合三套菜单
4. 自动合并多道菜食材成买菜清单
5. 生成详细做法（新手版/快手版/少油版/儿童版/老人版）
6. 生成整桌做饭顺序（产品亮点）
7. 生成替换建议
8. 生成解释文案

### 12.2 不适合做的

1. 真实价格（应来自历史表 + 公开数据）
2. 实时库存（不要编）
3. 高风险饮食建议（孕妇、糖尿病等 → 提示看医生）
4. 自由输出文本（必须严格 JSON）

## 17. 大模型 Prompt 设计

### 17.1 生成吃什么方案 Prompt

```text
你是家庭买菜做饭助手。
你的任务不是推荐单个菜，而是根据家庭情况生成三套完整晚餐方案。

用户需求：
- 城市：深圳
- 人数：{{people_count}}
- 成人：{{adults}}
- 儿童：{{children}}
- 老人：{{elders}}
- 预算：{{budget}} 元
- 口味：{{taste}}
- 忌口：{{avoid}}
- 做饭时间：{{time_limit}} 分钟
- 买菜方式：{{shopping_channel}}
- 厨具：{{kitchen_tools}}

今日推荐食材：
{{daily_ingredient_pool}}

候选菜品：
{{candidate_dishes}}

要求：
1. 输出严格 JSON，不要输出多余解释。
2. 生成三套方案：省钱快手型 / 营养均衡型 / 孩子老人友好型（或改善伙食型）。
3. 每套包含 3-5 道菜。
4. 每套必须至少包含 1 道蔬菜。
5. 有儿童时，必须至少包含 1 道儿童友好菜。
6. 避开用户忌口。
7. 尽量不超过预算。
8. 买菜清单要合并重复食材。

```

## 25. 项目最关键判断

### 25.1 方向成立

解决的是**前置决策问题**：不是"我想学一道菜"，而是"我根本不知道今晚买什么、做什么、怎么做"。

### 25.2 不要做错方向

不要变成普通菜谱大全。真正重点不是菜谱数量，而是：

```text
根据家庭情况生成一桌饭
根据深圳季节和常见食材推荐
买菜清单可执行
做饭顺序可执行
用户不用思考
```

### 25.3 第一版成功标准

```text
用户看完三套菜单后，觉得其中一套真的可以今晚照着买
买菜清单没有明显漏项
做法新手能看懂
做饭顺序有帮助
预算大体合理
没有推荐明显买不到的菜
```

### 25.4 最终产品核心

> **用菜品库 + 时令数据 + 去年同期价格 + 大模型，把"今天不知道买什么菜"变成"三套可直接照做的家庭晚餐方案"。**

---

## 26. 最推荐的 MVP 方案

- **产品名**：今天吃什么
- **副标题**：帮你想好买什么、做什么、怎么做
- **技术**：Next.js + Supabase + DeepSeek API
- **数据**：300 道菜 + 200 食材 + 深圳时令表 + 去年同期价格 + 平台搜索词
- **功能**：输入家庭情况 → 吃什么方案 → 买菜清单 → 详细做法 → 做饭顺序 → 替换 + 用户反馈
- **暂不做**：实时商超、商城、支付、复杂营养、多城市

---

## 27. 一句话总控思路

```text
先用大模型批量生成菜品库和食材库；
再用深圳/华南时令表和去年同期公开菜价做基础推荐；
今年只用少量价格修正和用户反馈校准；
前端让用户输入家庭情况；
后端用规则筛选菜品；
大模型负责生成多套菜单、买菜清单、做饭顺序和详细做法；
第一版不接沃尔玛/朴朴实时接口，只提供平台搜索词和常见可买判断。
```
